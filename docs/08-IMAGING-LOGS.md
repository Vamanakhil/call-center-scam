# 08 — Imaging Logs

## 8.1 What is forensic imaging?

A bit-for-bit copy of a storage device (HDD/SSD/USB/SD/phone/server volume) that:
- copies every sector including deleted, slack, unallocated
- preserves file-system metadata
- is verifiable by cryptographic hash
- is the **only** copy to be analysed — the original is preserved as evidence

## 8.2 Imaging formats

| Format | Ext | Tool | Notes |
|---|---|---|---|
| **E01 / Ex01** | `.E01` | EnCase / FTK Imager | Industry standard; compressed, segmented, built-in hash. **Used in this lab.** |
| **DD / Raw** | `.dd` | `dd`, FTK Imager | Uncompressed; 2× source size; portable. |
| **AFF4** | `.aff4` | AFF4 tools | Modern open standard. |
| **AD1** | `.AD1` | FTK | Logical-only; selective exports, not full disk. |

## 8.3 The imaging log

Every imaging operation produces a log. The log must be:
1. generated by the imaging tool (FTK Imager writes a `.txt` automatically)
2. **printed and signed** by the operator and a witness
3. appended to the case diary
4. stored alongside the image in the same evidence locker

## 8.4 Step-by-step FTK Imager procedure (full disk)

```
1. Connect the write-blocker (Tableau T8 USB or WiebeTech USB)
2. Connect the source drive to the write-blocker
3. Connect the write-blocker to the examiner workstation
4. Open FTK Imager 4.7 → I免N4.7.1.2 x64
5. File → Create Disk Image
6. Source : Physical Drive → select \\.\PHYSICALDRIVE0 (or the USB)
7. Click Next
8. Image Destination : E:\EVIDENCE\AGENT-01.E01
9. Image Type : E01
10. Segment size : 4096 MB (4 GB — fits on any FAT32/exFAT USB)
11. Compression : 6 (Best)
12. Check "Create a directory listing of the source drive"
13. Check "Verify images after they are created"  ← CRITICAL
14. Click Start
15. Wait. For a 1 TB drive on USB 3.0 with compression: 2-3 hours.
16. When complete, FTK shows "Image created successfully" and a hash.
17. Copy the .E01, .E02, .E03... and the .txt log to a SECOND destination.
18. Both copies are sealed separately as evidence.
19. Print the log, sign it, attach to the case diary.
```

## 8.5 Per-device-class imaging notes

### 8.5.1 Workstation (HDD)
- Use a **physical** image (not logical) to capture deleted files, slack, unallocated.
- Compression level 6 gives 40-60 % space savings with negligible speed penalty.
- 1 TB HDD @ 100 MB/s USB = ~3 hours.

### 8.5.2 SSD
- **Critical:** an SSD can perform **TRIM** in the background, destroying deleted data. Image the SSD **immediately** on seizure, even before RAM.
- If the SSD supports encryption (e.g., BitLocker, Opal), the encryption keys are in RAM — capture RAM first, then image.

### 8.5.3 USB
- Image via write-blocker; logical image is usually sufficient (no slack/recycle to worry about).
- For pen drives, capture the **full physical** image to include the bad-block remapping area.

### 8.5.4 Server (CRM-SERVER)
- **Do not pull the plug.** Use a controlled shutdown so open files flush.
- Image all disks in sequence (or in parallel via a hardware duplicator).
- 2 TB × 2 disks in RAID-1 = 2 TB to image per disk. Expect 6-8 hours.
- Capture a **live MySQL dump** first (see SOP §6.6) so the data is independent of the disk image.

### 8.5.5 Mobile phone
- Standard forensic image is not possible without a tool like **Cellebrite UFED** or **MSAB XRY**.
- Logical extraction (contacts, SMS, call log, app data) takes 30-90 minutes per phone.
- Physical extraction (bit-for-bit of the flash) is hardware-specific and may take 4-12 hours.
- For the **burner Android** in this lab, use UFED or chip-off (advanced — instructor demo only).

### 8.5.6 Router / Switch
- Not a disk image — use **console cable** + `show running-config` + `copy running-config tftp`.
- Photograph the boot process, the admin web UI, and the WAN/LAN MAC stickers.

## 8.7 Common mistakes

- ❌ Imaging a live system to a **logical** path (skipping physical) — you lose deleted data and slack space.
- ❌ Skipping the verify step to save time. Always verify.
- ❌ Storing the image on the same drive as the source. Always on a separate physical disk.
- ❌ Using a USB 2.0 write-blocker for a 1 TB drive — 12 hours wasted. Use USB 3.0.
- ❌ Hashing only the first .E01 segment. The whole image is the concatenation of all segments.
- ❌ Forgetting to capture the FTK log text file (it doesn't auto-copy with the .E01).
